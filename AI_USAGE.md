### Fix: dominio User dependía de crypto (violación de Dependency Rule)
- **Herramienta de IA utilizada:** Claude Code (Modelo Claude Sonnet 5)
- **Prompt o instrucción proporcionada:** El usuario señaló que `Password` no puede hashear en el dominio (import de `crypto`) ni `UserId` puede autogenerar un `randomUUID`: el dominio no puede importar librerías. `Password` solo debe validar los mínimos de caracteres; el hasheo debe hacerse en el caso de uso que orquesta la creación. `UserId` debe recibir siempre su valor desde el componente que inicializa el usuario (use case) y solo validar que sea numérico y de máximo 12 caracteres.
- **Resultado obtenido:** Se eliminó el import de `crypto` en `password.ts` y `user-id.ts`. `Password` ahora expone `ensureIsStrong(plainText)` (valida, no hashea) y `fromHash(hash)` (reconstituye); se quitó `create()` y `matches()`. `UserId` ahora expone un único `create(value)` que valida `/^\d+$/` y longitud <= 12; se quitó la generación interna de UUID. `User.register()` pasó a recibir `id: UserId` como parámetro en vez de generarlo. Se reescribieron los 3 test suites afectados acorde a la nueva API. 20 tests en 5 suites, todos en verde.
- **Modificaciones realizadas por el equipo:** (Pendiente de revisión humana)
- **Lecciones aprendidas o limitaciones identificadas:** La justificación anterior ("`crypto` es built-in de Node, no un paquete npm, por lo tanto no viola la regla de dependencias") era incorrecta: la Dependency Rule de Clean Architecture prohíbe que el dominio dependa de CUALQUIER mecanismo técnico/de infraestructura (hashing, generación de IDs, IO, fecha/hora del sistema, etc.), sea o no built-in del runtime. Esas responsabilidades pertenecen a la capa de Use Cases/Infraestructura (p. ej. un futuro `IPasswordHasher` y un generador de IDs inyectado desde el `RegisterUserUseCase`).

### Fix: verificación de JWT duplicada entre ProgressController y SecureCommandDecorator

- **Herramienta de IA utilizada:** Claude Code (Modelo Claude Sonnet 5)
- **Prompt o instrucción proporcionada:** El usuario señaló 3 reglas (aparentemente de una rúbrica externa) sobre el diseño de los decorators de seguridad: (1) el `token` no debe vivir en el constructor de un decorator tratado como singleton — ya se cumplía, porque la factory arma una instancia nueva por request; (2) el JWT no debe entrar como campo del Command — ya se cumplía, `SyncProgressCommand` solo tiene `userId: string`; (3) no debe duplicarse la verificación criptográfica del token entre el Controller y el Decorator, porque eso rompe la separación de responsabilidades de AOP — el Decorator debe ser el único responsable de verificar automáticamente la sesión. Se refactorizaron `SecureCommandDecorator`/`SecureQueryDecorator` para depender de `CurrentUserProvider` en vez de `(tokenGenerator, token)` sueltos. Se eliminó `auth.helper.ts` (su única función, `verifyAuth`, quedó absorbida dentro de `CurrentUserProvider`). Se actualizó `tokens.ts`: los tipos `SecureCommandServiceFactory`/`SecureQueryServiceFactory` ahora reciben `CurrentUserProvider` en vez de un `token: string`, y se agregó `CurrentUserProviderFactory` + su token `CURRENT_USER_PROVIDER_FACTORY` (construye el provider una vez por request a partir del Bearer token). Se actualizaron los 3 Controllers protegidos (`LevelController`, `ProgressController`, `LeaderboardController`) para construir un único `CurrentUserProvider` por request y pasarlo tanto a la factory de servicio (gate de seguridad) como, en el caso de `ProgressController`, a `getUserId()` para armar el Command — ambos usos comparten la misma instancia, por lo que la verificación criptográfica real ocurre como máximo una vez por request. `tsc --noEmit` sin errores.
